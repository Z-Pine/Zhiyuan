# 数据采集方案设计

## T020: 数据采集方案设计

## 一、数据源分析

### 1.1 院校数据

**官方数据源**:
- 教育部阳光高考平台: https://gaokao.chsi.com.cn
- 各省教育考试院官网
- 各高校招生网

**采集内容**:
- 院校基本信息（名称、代码、层次、类型）
- 院校标签（985/211/双一流）
- 地理位置、联系方式
- 招生简章、专业介绍

### 1.2 专业数据

**数据源**:
- 教育部本科专业目录
- 各高校专业设置
- 学科评估结果

**采集内容**:
- 专业名称、代码
- 学科门类、专业类
- 培养目标、核心课程
- 就业方向

### 1.3 录取分数数据

**数据源**:
- 各省教育考试院公布的历年录取数据
- 高校招生网公布的分数线
- 第三方教育数据平台

**采集内容**:
- 院校-专业-省份-年份-科类-批次
- 最低分、平均分、最高分
- 最低位次、招生人数

### 1.4 行业就业数据

**数据源**:
- 国家统计局
- 人社部就业数据
- 招聘网站（智联招聘、前程无忧）
- 行业研究报告

**采集内容**:
- 行业发展趋势
- 薪资水平
- 就业率
- 热门岗位

## 二、采集策略

### 2.1 采集方式

| 数据类型 | 采集方式 | 频率 | 优先级 |
|---------|---------|------|--------|
| 院校基础信息 | API + 爬虫 | 每年1次 | P0 |
| 专业数据 | 官方文件导入 | 每年1次 | P0 |
| 录取分数 | 爬虫 | 每年6-8月 | P0 |
| 行业数据 | API + 手动整理 | 每季度 | P1 |

### 2.2 爬虫设计

**技术选型**:
- 语言: Node.js / Python
- 框架: Puppeteer / Playwright / Scrapy
- 存储: PostgreSQL

**反爬虫策略**:
1. 请求频率控制（1-3秒/请求）
2. User-Agent轮换
3. IP代理池
4. Cookie/Session管理
5. 验证码识别（打码平台）

**数据清洗**:
1. 去重（基于院校+专业+年份）
2. 异常值检测
3. 格式标准化
4. 缺失值处理

### 2.3 数据更新策略

**增量更新**:
- 新数据：直接插入
- 变更数据：更新并记录版本
- 删除数据：软删除标记

**数据校验**:
- 完整性检查
- 一致性检查
- 合理性检查（分数范围、位次逻辑）

## 三、采集实施计划

### 3.1 第一阶段：核心数据（已完成）

**已完成**:
- ✅ 700+专业基础数据
- ✅ 院校基本信息框架
- ✅ 模拟录取分数数据

### 3.2 第二阶段：重点省份（T021）

**目标**: 采集5-10个重点省份的完整数据

**优先省份**:
1. 湖北（项目起始省份）
2. 北京（首都，院校集中）
3. 上海（经济中心）
4. 江苏（教育大省）
5. 浙江（新高考改革）
6. 广东（人口大省）
7. 四川（西部代表）
8. 山东（高考大省）

**每个省份采集内容**:
- 所有本科院校（100-150所）
- 所有本科专业（500-700个）
- 近3年录取分数数据

**预计工时**: 16小时

### 3.3 第三阶段：全国覆盖

**目标**: 覆盖全国31个省市自治区

**分批实施**:
- 第一批：重点省份（已完成）
- 第二批：其他省份（每批5-8个）
- 第三批：港澳台及特殊招生

## 四、数据质量保证

### 4.1 准确性保障

1. **多源验证**: 同一数据从多个来源采集对比
2. **人工抽检**: 定期人工核查数据准确性
3. **用户反馈**: 建立数据纠错机制

### 4.2 完整性保障

1. **采集监控**: 实时监控采集进度和成功率
2. **异常告警**: 数据缺失或异常时自动告警
3. **补采机制**: 失败任务自动重试

### 4.3 时效性保障

1. **定期更新**: 每年高考季后全面更新
2. **实时更新**: 重要政策变化及时更新
3. **版本管理**: 保留历史数据版本

---

## 五、法律合规

### 5.1 数据采集规范

1. **遵守robots.txt**: 尊重网站爬虫协议
2. **合理频率**: 避免对目标网站造成压力
3. **数据使用**: 仅用于教育服务，不用于商业目的

### 5.2 数据安全

1. **数据加密**: 敏感数据加密存储
2. **访问控制**: 严格的数据访问权限
3. **备份策略**: 定期备份，防止数据丢失
